# 本地票据 OCR 识别服务 — 技术文档

**文档版本**：v1.0  
**更新日期**：2026-06-15  
**代码基线**：invoice 项目（基于 chineseocr）  
**运行环境**：Python 3.5/3.6，本地 CPU/GPU

---

## 1. 系统概述

### 1.1 技术定位

本地部署的票据 OCR 微服务：**Flask 提供 HTTP 接口**，**YOLOv3 做文字区域检测**，**CRNN + CTC 做文字识别**，**正则规则做字段结构化**。推理完全在本地完成，无外部 OCR API 调用。

### 1.2 技术栈

| 层级 | 技术 |
|------|------|
| Web 服务 | Flask + flask-cors |
| 文字检测 | YOLOv3（Keras / OpenCV DNN / Darknet，默认 Keras） |
| 文字识别 | CRNN（PyTorch 或 Keras，默认 `ocr-lstm.pth`） |
| 方向检测 | OpenCV DNN（`Angle-model.pb`） |
| 图像处理 | OpenCV、PIL、NumPy |
| PDF 支持 | pdf2image + Poppler |
| 深度学习框架 | TensorFlow 1.x（检测）、PyTorch（识别） |

### 1.3 模型文件（`models/`）

| 文件 | 用途 |
|------|------|
| `text_type.h5` | 发票类型区域检测 |
| `text_electronic.h5` | 电子普票文字区域检测 |
| `text_machine.h5` | 机打发票文字区域检测 |
| `ocr-lstm.pth` | 中文 CRNN 识别（默认） |
| `ocr-dense.pth` / `ocr-english.pth` | 备选识别模型 |
| `Angle-model.pb` | 文字方向检测 |

## 2. 现状架构

### 2.1 请求处理流程

```
POST /invoice-ocr
  → 保存上传文件（test/）
  → PDF 则 pdf_to_jpg（第一页，dpi=200）
  → remove_stamp（红通道阈值去红章）
  → Recognition_invoice（类型判断）
       → model_post_type.ocr（text_type.h5 检测 + CRNN）
       → 关键词：电子普通 / 普通发票 / 专用发票
       → 返回 1（电子普票）或 2（机打票）
  → ocr_E 或 ocr_M（对应检测模型 + CRNN）
  → invoice_e 或 invoice_m（正则抽字段）
  → build_api_result JSON
```

### 2.2 核心模块

| 模块 | 路径 | 职责 |
|------|------|------|
| 入口 | `app.py` | 路由、上传、类型分支、响应 |
| 配置 | `config.py` | GPU、模型路径、检测引擎开关 |
| 类型 OCR | `model_post_type.py` | 类型检测 + CRNN |
| 电子票 OCR | `model_postE_invoice.py` | 电子普票检测 + CRNN |
| 机打票 OCR | `model_postM_invoice.py` | 机打票检测 + CRNN |
| 电子票解析 | `application/invoice_e.py` | 发票代码/号码/日期/金额/校验码 |
| 机打票解析 | `application/invoice_m.py` | 同上，规则略有差异 |
| 检测实现 | `text/keras_detect*.py` | YOLO 文字框 |
| 识别实现 | `crnn/crnn_torch.py` | CRNN 推理 |
| 图像工具 | `apphelper/image.py` | 框合并、排序、旋转裁剪等 |

### 2.3 类型判断逻辑（现状）

位置：`app.py` → `Recognition_invoice`

```python
type_1 = re.findall('电子普通', txt)
type_2 = re.findall('普通发票', txt)
type_3 = re.findall('专用发票', txt)
if len(type_1) > 0:
    return 1   # → ocr_E + invoice_e
else:
    return 2   # → ocr_M + invoice_m
```

- 类型 OCR 输入为**去红章后的 RedThresh 图**
- 仅「电子普通」走电子普票分支，其余（含专票、普票、无匹配）均走机打分支

### 2.4 字段解析（现状）

**电子普票**（`application/invoice_e.py`）示例规则：

- 发票代码：`代码:\d*` / `代码\d*`
- 发票号码：`号码:\d*` / `号码\d*`
- 开票日期：`日期:...年...月...日` 或 `日期[0-9]{8}`
- 税后金额：`￥[0-9]{1,8}.[0-9]{1,2}`
- 校验码：`校验码:[0-9]{1,20}`

**机打票**（`application/invoice_m.py`）使用数字长度等启发式规则。

### 2.5 本地算力配置

`config.py`：

```python
GPU = False          # True 时启用 GPU
GPUID = 0
ocrFlag = 'torch'    # CRNN 引擎：torch / keras
opencvFlag = 'keras' # 检测引擎：keras / opencv / darknet
DETECTANGLE = True   # 文字方向检测
LSTMFLAG = True      # 使用 ocr-lstm.pth
```

- `GPU = False` 时设置 `CUDA_VISIBLE_DEVICES=''`，NMS 降级为 `cython`
- 有 NVIDIA GPU 时可安装 `tensorflow-gpu` 并设 `GPU = True`

## 3. 能力边界与根因分析

### 3.1 组件可复用性

| 组件 | 可复用性 | 说明 |
|------|----------|------|
| CRNN（`ocr-lstm.pth`） | 高 | 通用中文 OCR，不限票种 |
| Flask + PDF 转图 | 高 | 与票种无关 |
| 检测后处理 | 高 | TextDetector、sort_box 等 |
| 文字检测 YOLO | 中 | 仅在增值税发票上训练，新票种可能漏检 |
| 类型判断 | 低 | 硬编码关键词 |
| 字段解析 | 低 | 仅发票字段 |
| 去红章 | 低 | 仅适用于发票场景，可能伤害其他票据 |

### 3.2 无法识别电子行程单等技术原因

1. **无行程单类型分支**：关键词不匹配「电子行程单」
2. **检测模型未覆盖行程单版式**：YOLO 训练样本为发票
3. **解析器无对应字段**：无旅客姓名、航班号、客票号等规则
4. **去红章可能破坏图像**：非发票不应使用该预处理

### 3.3 识别为空（code 104）触发条件

`app.py` 中当 `len(res) == 0` 时返回 104。  
`res` 来自 parser 的 `self.res` 字典；若 OCR 有文本但正则均未命中，则 fields 为空 → 104。

## 4. 目标架构

### 4.1 目标流水线

```
上传
  → 预处理（PDF 转图；按 docType 可选去红章）
  → 统一 OCR（检测模型 + CRNN → lines[]）
  → DocRouter（关键词 / 规则 → docType）
  → ParserRegistry（按 docType 选解析器）
  → 统一 JSON（docType + fields + rawLines）
```

### 4.2 目标模块结构

```
invoice/
├── app.py                      # 编排，不再硬编码 if/else
├── config.py                   # 扩展 DOC_TYPES 配置表
├── model_post.py               # 统一 OCR 入口（从 E/M 抽取）
├── application/
│   ├── doc_router.py           # 票种路由
│   └── parsers/
│       ├── __init__.py         # 解析器注册表
│       ├── vat_e.py            # 自 invoice_e 迁入
│       ├── vat_m.py            # 自 invoice_m 迁入
│       ├── digital_vat.py      # 数电票
│       ├── itinerary.py        # 电子行程单
│       ├── train.py            # 火车票
│       └── taxi.py             # 出租车
├── model_post_type.py          # 保留或合并进 model_post
├── model_postE_invoice.py      # 逐步 deprecated，逻辑迁入 model_post
├── model_postM_invoice.py
└── docs/
    ├── 产品文档.md
    └── 技术文档.md
```

### 4.3 票种路由规则（初版）

| docType | 关键词 | 检测模型 | 去红章 | Parser |
|---------|--------|----------|--------|--------|
| `vat_e` | 电子普通、增值税电子普通 | `text_electronic.h5` | 是 | `vat_e` |
| `vat_m` | 普通发票、专用发票 | `text_machine.h5` | 是 | `vat_m` |
| `digital_vat` | 数电、电子发票（ | `text_electronic.h5` | 否 | `digital_vat` |
| `itinerary` | 电子行程单、行程单、ITINERARY | `text_electronic.h5` | 否 | `itinerary` |
| `train` | 铁路电子客票、火车票 | `text_electronic.h5` | 否 | `train` |
| `taxi` | 出租车、出租汽车 | `text_electronic.h5` | 否 | `taxi` |
| `unknown` | 无匹配 | `text_electronic.h5` | 否 | 无（仅 rawLines） |

路由建议在**原图 OCR 结果**上执行，避免去红章影响类型判断。

## 5. OCR 流水线细节

### 5.1 单张识别流程（`model_postE_invoice.model`）

1. `eval_angle`：方向检测，必要时旋转 90/180/270
2. `text_detect`：YOLO 出文字框 → TextDetector 合并行
3. `sort_box`：行排序
4. `crnnRec`：逐框 CRNN 识别
5. 返回 `[{cx, cy, text, w, h, degree}, ...]`

### 5.2 关键检测参数

电子票 OCR（`model_postE_invoice.ocr`）当前参数：

```python
MAX_HORIZONTAL_GAP=50
MIN_V_OVERLAPS=0.6
MIN_SIZE_SIM=0.6
TEXT_PROPOSALS_MIN_SCORE=0.1   # 较低，提高召回
TEXT_PROPOSALS_NMS_THRESH=0.3
TEXT_LINE_NMS_THRESH=0.7
leftAdjust=True
rightAdjust=True
alph=0.1
detectAngle=True
```

新票种若漏检，可优先尝试降低 `TEXT_PROPOSALS_MIN_SCORE`，再考虑微调 YOLO。

### 5.3 PDF 处理

- 依赖 Poppler（Windows 可通过 conda 安装或设置 `POPPLER_PATH`）
- 仅转换第一页，`dpi=200`
- 实现：`app.py` → `pdf_to_jpg` / `_find_poppler_path`

### 5.4 中文路径

Windows 下 OpenCV 无法直接 `imread` 中文路径，使用：

- `imread_unicode`：`np.fromfile` + `cv2.imdecode`
- `imwrite_unicode`：`cv2.imencode` + `tofile`

## 6. 解析器设计

### 6.1 解析器接口约定

```python
class BaseParser:
    def __init__(self, ocr_lines: list):
        """
        ocr_lines: [{'text': str, 'cx': float, 'cy': float, ...}, ...]
        """
        self.lines = ocr_lines
        self.res = {}

    def parse(self) -> dict:
        """返回 fields 字典"""
        raise NotImplementedError
```

### 6.2 注册表

```python
# application/parsers/__init__.py
PARSERS = {
    'vat_e': VatEParser,
    'vat_m': VatMParser,
    'digital_vat': DigitalVatParser,
    'itinerary': ItineraryParser,
    # ...
}

def get_parser(doc_type, ocr_lines):
    cls = PARSERS.get(doc_type)
    if cls is None:
        return None
    return cls(ocr_lines).parse()
```

### 6.3 电子行程单解析规则示例

| 字段 | 正则 / 逻辑 |
|------|-------------|
| 旅客姓名 | `旅客姓名[:：]\s*(\S+)` |
| 航班号 | `[A-Z]{2}\d{3,4}` |
| 客票号 | `\d{13,15}` |
| 金额 | `合计\|票价\|金额` 邻近金额模式 |

### 6.4 数电票解析注意点

- 字段标签与旧增值税票不同（如「发票号码」格式、无「发票代码」）
- 需单独样本集验证，不可复用 `invoice_e` 规则
- 项目 test 目录已有 `dzfp_*.pdf` 可作为首批样本

## 7. API 技术规范

### 7.1 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/` | Web 测试页（`templates/index.html`） |
| GET | `/health` | `{"status": "ok"}` |
| POST | `/invoice-ocr` | 主识别接口 |

### 7.2 请求

```
Content-Type: multipart/form-data
Body: file=<binary>
```

允许扩展名：`jpg`, `jpeg`, `png`, `pdf`（定义于 `app.py` → `allowed_extension`）

### 7.3 响应（目标格式）

```json
{
  "code": 100,
  "message": "识别成功",
  "data": {
    "docType": "vat_e",
    "docTypeName": "增值税电子普通发票",
    "fields": {
      "发票代码": "xxx",
      "发票号码": "xxx"
    },
    "rawLines": []
  },
  "FileName": "invoice.pdf",
  "ocrIdentifyTime": "2026-06-15 16:00:00"
}
```

**成功判定（建议改造后）**

- `fields` 非空 → code 100
- 或 `docType == 'unknown'` 且 `rawLines` 非空 → code 100
- OCR 完全无文本 → code 104

### 7.4 可选扩展端点

```
POST /ocr-raw
```

仅返回 OCR 行，不做结构化，便于调试与新票种标注。

## 9. 模型训练与优化（可选阶段）

### 9.1 何时需要微调 YOLO

- 某票种 raw OCR 漏检率 > 30%
- 降低 `TEXT_PROPOSALS_MIN_SCORE` 后仍大量漏字

### 9.2 微调流程

1. LabelImg / CVAT 标注文字框（每类 100–300 张）
2. 基于 `text/keras_yolo3.py`、`text/keras_detectE_invoice.py` 微调
3. 新权重如 `models/text_general.h5` 或 `text_itinerary.h5`
4. 在 `config.DOC_TYPES` 中指向新权重

### 9.3 CRNN

- 默认**不重训**，中文通用
- 纯英文票据可切换 `ocr-english.pth`（`config.chinsesModel = False`）

### 9.4 LLM Fallback（可选）

```
OCR lines → 拼接全文 → LLM + JSON Schema → fields
```

触发条件：Parser 返回空且 docType 非 unknown。部署可为内网 API，OCR 仍在本地。

## 10. 部署与运维

### 10.1 启动

```bash
pip install -r requirements.txt
# 确保 models/ 已放置模型权重
```

## 12. 风险与技术对策

| 风险 | 对策 |
|------|------|
| TensorFlow 1.x 与 Python 版本老旧 | 维持现有环境；大版本升级单独立项 |
| 多模型内存占用 | 懒加载或共享 CRNN session |
| Web 并发 | 生产用 WSGI + 多 worker；注意 TF session 线程安全 |
| Windows 中文路径 | 已用 imdecode，新代码继续沿用 |
| 类型关键词冲突 | 路由规则按优先级排序，长关键词优先 |
