# 本地票据 OCR 识别服务 — 产品文档

**文档版本**：v1.0  
**更新日期**：2026-06-15  
**产品名称**：本地票据 OCR 识别服务（invoice 项目）  
**当前基线**：支持增值税电子普通发票、增值税普通发票、增值税专用发票

---

## 1. 产品概述

### 1.1 产品定位

基于开源 chineseocr 架构的**本地部署**票据识别服务。用户通过 Web 页或 HTTP 接口上传图片/PDF，系统在本地完成文字检测与识别，返回结构化字段，**不依赖云端 OCR API**。

### 1.2 核心价值

| 价值点 | 说明 |
|--------|------|
| 数据本地化 | 票据不上传第三方，适合内网与合规场景 |
| 零 API 费用 | 算力消耗在本机 CPU/GPU |
| 可扩展 | 架构可逐步接入更多票种 |
| 轻量集成 | Flask HTTP 接口，易于对接 ERP / 费控系统 |

### 1.3 运行方式

- 启动：`python app.py`（默认端口 `11111`）
- 识别接口：`POST /invoice-ocr`，表单字段 `file`
- 支持格式：jpg、jpeg、png、pdf（PDF 识别第一页）

## 2. 目标用户与使用场景

### 2.1 目标用户

- **财务人员**：报销审核、票据归档
- **系统集成方**：通过 API 对接企业内部系统
- **开发测试人员**：通过 Web 测试页验证识别效果

### 2.2 核心场景

| 场景 | 当前表现 | 产品期望 |
|------|----------|----------|
| 上传增值税发票 | 可识别主要字段 | 保持能力不退化 |
| 上传航空电子行程单 | 识别失败或字段错误 | 识别票种并抽取核心字段 |
| 上传数电票（dzfp） | 不稳定 | 支持数电票主要字段 |
| 上传未知票据 | 返回「识别为空」 | 返回 OCR 全文供人工或下游处理 |
| API 批量调用 | 仅返回发票字段 | 统一返回票种 + 字段 + 原文 |

## 3. 当前能力与边界

### 3.1 已支持票种

| 票种 | 识别字段 |
|------|----------|
| 增值税电子普通发票 | 发票代码、发票号码、开票日期、校验码、税后金额 |
| 增值税普通发票 | 同上（机打票解析规则） |
| 增值税专用发票 | 同上（机打票解析规则） |

### 3.2 不支持的内容（现状）

- 航空**电子行程单**
- **数电票**（新版全电发票）
- 火车票、出租车票、定额发票等
- 非票据类图片

### 3.3 不支持的原因（产品视角）

本产品从设计之初仅面向**增值税发票**：

1. 类型判断只识别「电子普通 / 普通发票 / 专用发票」等关键词
2. 检测与解析规则按发票版式与字段命名编写
3. 电子行程单、数电票等版式与字段体系完全不同，无法直接套用

> **说明**：「电子行程单」≠「增值税电子发票」，二者属于不同业务体系，不可混为一谈。

---

## 4. 产品目标与规划

### 4.1 总体目标

在**不替换现有本地 OCR 架构**的前提下：

1. 保持增值税发票识别能力不变
2. 逐步支持更多常见报销凭证
3. 未知票种提供 OCR 全文兜底，避免直接失败
4. 统一 API 响应，便于前端与第三方集成

### 4.2 非目标（本期不做）

- 不追求全品类票据 100% 覆盖
- 不强制依赖云端 OCR 或 LLM（可作为可选增强）
- 不提供移动端 SDK（维持 HTTP 接口）
- 不替换现有 CRNN 识别引擎

### 4.3 票种优先级

| 优先级 | 票种 | 计划迭代 |
|--------|------|----------|
| P0 | 增值税发票（现有三类） | 基线，持续回归 |
| P0 | 未知票据 OCR 兜底 | 迭代 1 |
| P1 | 数电票 | 迭代 2 |
| P1 | 电子行程单 | 迭代 3 |
| P2 | 火车票、出租车票 | 迭代 4 |
| P3 | 其他票种 | 按样本量排期 |

### 4.4 各票种目标字段（V1）

**增值税发票（现有）**  
发票代码、发票号码、开票日期、校验码、税后金额

**数电票**  
发票号码、开票日期、价税合计、购买方、销售方、税额

**电子行程单**  
旅客姓名、航班号、客票号、出发/到达、航班日期、票价/合计

**火车票**  
乘车人、车次、出发站、到达站、乘车日期、票价

**未知票种**  
无结构化字段，返回 OCR 全文行列表

| 功能 | 说明 |
|------|------|
| 票种自动识别 | 返回 `docType`、`docTypeName` |

| 可选 raw 模式 | 仅返回 OCR 行，不做结构化（按需） |

## 6. 接口与交互

### 6.1 请求

```
POST /invoice-ocr
Content-Type: multipart/form-data
参数：file（jpg / jpeg / png / pdf）
```

### 6.2 响应结构（规划统一格式）

```json
{
  "code": 100,
  "message": "识别成功",
  "data": {
    "docType": "itinerary",
    "docTypeName": "电子行程单",
    "fields": {
      "旅客姓名": "张三",
      "航班号": "CA1234"
    },
    "rawLines": [
      { "text": "电子行程单", "cx": 0, "cy": 0 }
    ]
  },
  "FileName": "xxx.pdf",
  "ocrIdentifyTime": "2026-06-15 16:00:00"
}
```

### 6.3 状态码

| code | 含义 |
|------|------|
| 100 | 识别成功（有结构化字段，或 unknown 且有 rawLines） |
| 101 | 请求参数错误 |
| 102 | 文件格式不支持 |
| 103 | PDF 转换失败 |
| 104 | 识别为空（OCR 无任何文本） |
| 105 | 识别异常 |

### 6.4 兼容性承诺

- 增值税发票成功时，`fields` 键名与现网一致
- 旧客户端可忽略新增字段 `docType`、`rawLines`
- 不破坏现有 `/invoice-ocr` 路径与 `file` 参数名

## 7. 迭代计划

### 迭代 1：通用 OCR 兜底 + 架构骨架（第 1 周）

**交付**

- 插件化票种路由与解析器目录
- 未知票种返回 OCR 全文
- 增值税识别零回归

**验收**

- [ ] 增值税样本结果与改造前一致
- [ ] 行程单 PDF 不再 104，返回 rawLines
- [ ] 服务正常启动

---

### 迭代 2：数电票（第 2 周）

**交付**

- 数电票类型识别与字段解析
- 至少 20 张样本验证

**验收**

- [ ] 数电票核心字段召回率 ≥ 80%
- [ ] 旧增值税无退化

---

### 迭代 3：电子行程单（第 3 周）

**交付**

- 行程单类型识别与字段解析
- 禁用对行程单的去红章预处理

**验收**

- [ ] docType 正确为 itinerary
- [ ] 姓名、航班号、客票号召回 ≥ 75%

---

### 迭代 4：火车票 / 出租车 + 质量巩固（第 4 周）

**交付**

- 两类 parser
- 按票种组织测试样本
- 更新 README 支持范围说明

**验收**

- [ ] 每类 ≥ 10 张样本通过结构化或 raw 兜底
- [ ] 文档与 API 示例完整

---

### 迭代 5+（可选）

- **检测模型微调**：某票种 OCR 漏检严重时启动
- **LLM 字段抽取**：版式过多、规则维护成本过高时作为 fallback

## 8. 成功指标（KPI）

| 指标 | 基线 | 迭代 1 | 迭代 4 |
|------|------|--------|--------|
| 增值税识别成功率 | 现网水平 | 100% 不退化 | 100% 不退化 |
| 非发票完全失败率 (104) | 接近 100% | < 10% | < 5% |
| 数电票核心字段召回 | — | — | ≥ 80% |
| 行程单核心字段召回 | — | — | ≥ 75% |
| 单次识别耗时 (CPU) | 记录基线 | 增幅 < 15% | 增幅 < 20% |

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 去红章破坏非发票图像 | 行程单 OCR 变差 | 仅增值税类型启用去红章 |
| 类型关键词误判 | 字段解析错误 | 规则优先级 + 样本回归 |
| 数电票版式更新 | 规则失效 | 持续更新样本库 |
| 旧客户端不兼容 | 集成方投诉 | 保留 fields 键名，文档说明新字段 |
